4. **Process isolation** (Test 92): ExifTool failures don't crash the server — verified with corrupt files

### Tests Passing: 178/260
Total new tests this session: 17 (161 → 178)

### Architecture Changes
- tilde-photos crate fully implemented: 5 modules (exiftool, ingest, organize, thumbnail, watcher)
- File watcher (notify-rs) on _inbox/ and _library-drop/ with configurable debounce
- ExifTool subprocess integration for metadata read/write
- Organization pattern engine with date parsing and trip tag support
- Thumbnail pipeline using image crate (JPEG/PNG/WebP)
- Server startup processes existing inbox files + starts watcher
- Config: added watch_debounce_seconds, exiftool_timeout_seconds, ffmpeg_timeout_seconds
- Slimmed image crate (disabled AVIF/rav1e for disk/compile savings)

### Key Design Decisions
- ExifTool as subprocess (not library) for metadata — matches spec's process isolation requirement
- Watcher uses debounce HashMap + processor thread (not tokio async) since file processing is CPU-bound
- Non-media files in inbox are ignored by watcher; only media extensions trigger processing
- Corrupt files with valid magic bytes but no date → _untriaged (not _errors)
- _errors/ reserved for complete processing failures (file unreadable, etc.)

### Architecture Changes
- tilde-cal crate: Full CalDAV handler with auth, REPORT, sync-collection, getctag

- Migration 003: calendars, calendar_objects, addressbooks, contacts, sync_changes tables

### Key Design Decisions
- Built CalDAV/CardDAV directly (not via RustiCal) since RustiCal integration was too complex
- Auth check done inside CalDAV/CardDAV handlers (not middleware) since nest_service doesn't support state-typed middleware
- Sync-collection uses monotonic sync_token per calendar/addressbook
- Soft-delete for calendar objects and contacts (deleted=1 flag) to support sync-collection 404 responses
