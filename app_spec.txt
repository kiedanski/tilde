<project_specification>
  <project_name>tilde — Personal Cloud Server</project_name>

  <overview>
    A monolithic Rust binary serving open protocols (WebDAV, CalDAV, CardDAV, MCP, IMAP-as-client)
    that reuse existing mature client ecosystems rather than inventing new ones. Deployable on a
    cheap VPS (Hetzner, EC2) or behind a reverse proxy (Pangolin/Caddy) at home. Files are the
    source of truth, SQLite is a rebuildable cache, originals are immutable except for in-file
    metadata edits. CLI-first configuration, systemd-integrated installation, export-first
    architecture. Single-user. No web UI beyond a minimal Nextcloud Login Flow v2 page.

    Target user: a Linux-proficient individual who wants professional infrastructure they control
    and can migrate off of at any time. Target footprint: 1 vCPU, 256–512MB RAM, 15–30MB static
    musl binary. License: AGPL-3.0.
  </overview>

  <architecture>
    <overview>
      Monolithic axum + tokio binary. Single HTTPS listener on port 443 with path-based dispatch
      to per-protocol handlers. Cargo workspace with one crate per feature area.

      Request flow:
        Client → rustls TLS → axum Router → tower middleware (auth, logging, rate-limit)
          → path dispatch:
            /dav/*           → WebDAV handler (files, notes)
            /caldav/*        → RustiCal CalDAV handler
            /carddav/*       → RustiCal CardDAV handler
            /.well-known/*   → RFC 6764 well-known redirects
            /mcp/*           → rmcp Streamable HTTP handler
            /ocs/*           → Nextcloud OCS capabilities stub
            /login/v2/*      → Nextcloud Login Flow v2
            /remote.php/*    → Nextcloud compat aliases (redirect to /dav/*)
            /status.php      → Nextcloud status stub
            /metrics         → Prometheus metrics (localhost-only)
            /health          → Health check endpoint
    </overview>

    <sync_model>
      Last-writer-wins with Hybrid Logical Clocks (HLC) for metadata. ~500 LOC, no library dep.

      Per data type:
      - Calendars/contacts: ETag If-Match + RFC 6578 sync-token (handled by RustiCal)
      - Files (bytes): ETag If-Match; 412 Precondition Failed → client writes
        filename.sync-conflict-<date>-<device>.ext (Syncthing naming convention)
      - Photo metadata: LWW-per-column with HLC
      - Notes: file-level sync + conflict files on 412
      - Tracker rows: LWW + HLC + event log + Merkle tree for divergence detection

      No CRDTs in v1. Upgrade path to loro if collaborative editing becomes concrete need.
    </sync_model>

    <process_isolation>
      ExifTool, libheif decode, and ffmpeg run as subprocesses with:
      - Configurable timeout (default 30s for ExifTool, 60s for ffmpeg)
      - Memory limits via cgroup (where available)
      - Single-slot semaphore for thumbnail generation (prevents OOM on large HEIC)
      - Crashed subprocess cannot take down the server process
      - Errors logged and file moved to _errors/ directory
    </process_isolation>

  <core_features>

    <webdav_file_sync>
      <protocol>WebDAV Class 1 (RFC 4918) — NO Class 2 LOCK/UNLOCK</protocol>
      <description>
        Full WebDAV file serving with Nextcloud-compatible chunked upload support.
        Files stored on disk are the source of truth. SQLite indexes metadata for fast lookups.
      </description>

      <methods_supported>
        - OPTIONS (returns DAV: 1 header, supported methods)
        - GET (file download, supports Range headers)
        - PUT (file upload, atomic: write to tmp → fsync → rename → fsync parent dir)
        - DELETE (file/collection deletion)
        - MKCOL (create collection/directory)
        - MOVE (rename/move file or collection, updates oc:id mapping)
        - COPY (copy file or collection)
        - PROPFIND (Depth: 0 or 1, returns DAV properties)
        - PROPPATCH (set/remove custom properties)
        - HEAD (metadata without body)
      </methods_supported>

      <nextcloud_compatibility>
        - Chunked upload v2 protocol:
          1. Client creates upload session: MKCOL /dav/uploads/<user>/<session-id>/
          2. Client uploads chunks: PUT .../1, PUT .../2, etc.
          3. Client finalizes: MOVE session dir to destination path
          4. Server assembles chunks, validates size, returns final ETag
          5. Per-session size tracking, 24h session expiry, disk-space pre-check
          6. Streaming writes — never buffer full file in memory (supports 10GB+ videos)
        - Stable oc:id per file (server-assigned UUID surviving moves/renames)
        - oc:fileid, oc:permissions, oc:size custom properties
        - Login Flow v2 (POST /login/v2, POST /login/v2/poll — see auth section)
        - OCS capabilities stub at /ocs/v2.php/cloud/capabilities (returns minimal
          JSON declaring WebDAV chunked upload support)
        - /status.php returns {"installed":true,"version":"28.0.0",...} stub
        - /remote.php/dav/* aliases redirect to /dav/*

        These unlock: Nextcloud desktop sync clients (all platforms, with virtual-filesystem mode),
        Nextcloud Android/iOS apps, rclone, Joplin WebDAV, QOwnNotes, DAVx5.
      </nextcloud_compatibility>

      <conflict_handling>
        - All PUT/MOVE operations support If-Match ETag precondition
        - 412 Precondition Failed on ETag mismatch
        - Client expected to write conflict file: filename.sync-conflict-<YYYYMMDD-HHMMSS>-<device>.ext
        - Server never silently overwrites — always returns 412 if ETag doesn't match
      </conflict_handling>

      <finder_compat>
        - macOS Finder sends LOCK requests; respond with 200 OK (FakeLs pattern)
        - Finder is "works for casual browsing, not daily driver" — document this
        - Recommended mount methods: rclone (Linux), Mountain Duck (macOS), RaiDrive (Windows)
        - Nextcloud desktop sync client recommended as default for most users
      </finder_compat>
    </webdav_file_sync>

    <photos>
      <scope>~500 new photos/year + 30–50GB historical library import</scope>

      <storage_model>
        Content-addressed by UUID, not by hash (metadata mutability requires stable IDs even
        when file bytes change). Each photo record stores:
        - uuid: server-assigned, immutable, survives moves/renames
        - original_sha256: SHA-256 of original file at import (forensic dedup)
        - current_sha256: SHA-256 of current file (integrity verification, changes on metadata edits)
        - Blob path: blobs/by-id/<uuid>.<ext> with two-level directory sharding (first 2 + next 2 chars of UUID)
      </storage_model>

      <metadata_tagging>
        In-file metadata via XMP dc:Subject + IPTC Keywords, written by ExifTool subprocess.
        Rejected: XMP sidecar files (can separate from originals during export/move).

        Tradeoffs accepted:
        - SHA changes on every metadata edit (current_sha256 tracks this, original_sha256 is immutable)
        - Backup churn on tag edits
        - ~150ms per HEIC metadata write via ExifTool
        - ExifTool as system dependency

        Tag convention: soft prefix classifier recognizes:
        - trip:<name>      (e.g., trip:jamaica)
        - event:<name>     (e.g., event:wedding-anna)
        - person:<name>    (e.g., person:kids)
        - favorite         (no prefix)
        - Otherwise: plain XMP string, readable by Lightroom/digiKam/etc.
      </metadata_tagging>

      <organization>
        Configurable pattern (default: "{year}/{month:02}").
        Supported variables: {year}, {month:02}, {day:02}, {-trip} (from trip: tag).
        Example result: photos/2025/01-jamaica/IMG_1234.heic

        When metadata changes trigger a new organization path, server performs a WebDAV MOVE
        (not DELETE+PUT) to preserve oc:id and client sync state.

        manually_placed flag: if user moves a file manually (via WebDAV MOVE), set this flag
        to disable auto-organization for that file. Server won't re-sort it.
      </organization>

      <ingestion>
        No bulk-import CLI tool. Users process photos externally (ExifTool, Darktable, digiKam,
        custom scripts) then drop into the appropriate folder via WebDAV or local filesystem.

        Watched folder pipeline (notify-rs):
        1. File appears in _inbox/ or _library-drop/
        2. Debounce 5 seconds, verify file size is stable (not still being written)
        3. Validate file type (magic bytes, not just extension)
        4. Read metadata via ExifTool subprocess
        5. For _inbox/: apply organization pattern, move to final location
           For _library-drop/: preserve user's directory structure, bypass classifier
        6. Compute SHA-256 (original and current)
        7. Generate UUID, insert SQLite record
        8. Enqueue thumbnail generation (async, non-blocking)
        9. On any error: move file to _errors/ with error description sidecar .txt file

        _untriaged/ folder: photos with insufficient metadata for organization are moved here.
        User fixes metadata externally and re-drops into _inbox/.

        Encrypted file handling: Magic-byte detection (GPG, age, VeraCrypt, encrypted PDFs).
        Set content_readable = false, serve as opaque blob, skip thumbnail/indexing.
      </ingestion>

      <thumbnails>
        Two WebP thumbnails per photo:
        - 256px (square crop, for grid views)
        - 1920px (longest edge, for preview)
        Quality: configurable, default 80.

        Pipeline:
        - image crate + fast_image_resize (SIMD) for JPEG/PNG/WebP
        - libheif-rs for HEIC/HEIF decoding
        - ffmpeg subprocess for video thumbnails (first frame or configurable timestamp)
        - Blurhash string computed inline, stored in SQLite (for placeholder display)

        Storage: $CACHE_DIR/thumbnails/<uuid>/256.webp and 1920.webp
        Thumbnails are excluded from backup (regenerable from originals).
        Single-slot semaphore for concurrent thumbnail generation (prevents OOM on 48MP HEIC).
      </thumbnails>

      <mobile_sync>
        No Immich fork. Use existing WebDAV-speaking apps:
        - PhotoSync ($5, iOS/Android) — best option, WebDAV target, drops to _inbox/
        - FolderSync (Android, free) — WebDAV sync
        - Nextcloud mobile app — with "always available offline" on photos folder

        Offline browsing: Full library mirror on phone is feasible at this volume.
        Pattern: Nextcloud mobile app with offline-available + local gallery app
        (Aves Gallery on Android, Files app on iOS). Delete operations local; sync on reconnect.

    <mcp>
      <transport>Streamable HTTP only, via rmcp 1.5.x</transport>
      <endpoint>POST /mcp/ (single endpoint, Streamable HTTP spec)</endpoint>

      <auth>
        CLI-issued long-lived bearer tokens.
        Format: mcp_prod_<30 random alphanumeric bytes> (prefix for secret scanner detection).
        Storage: SHA-256 hash of token in SQLite (never store plaintext).
        Token creation: `tilde mcp token create --name "claude-desktop" --scopes "notes:read,calendar:read"`

        Scopes:
        - notes:read, notes:write
        - calendar:read, calendar:write
        - contacts:read, contacts:write
        - files:read, files:write
        - photos:read, photos:write
        - email:read
        - tasks:read, tasks:write
        - trackers:read, trackers:write
        - *:read (wildcard read)
        - Write scopes must always be listed explicitly (no *:write wildcard)

        Per-token rate limits (configurable, default 60 req/min).
        Token management: create, list, revoke, rotate.
      </auth>

      <oauth_stub>
        Server advertises OAuth via Protected Resource Metadata (RFC 9728) stub.
        Directs clients to use bearer tokens (no full OAuth 2.1 flow in v1).
        Skipped: PKCE, dynamic client registration, refresh tokens, consent UI, JWKS.
      </oauth_stub>

      </audit_log>

      <tools>
        ~15 tools, narrow and read-oriented by default:

        notes.search(query: string, limit?: int) → [{path, title, snippet, modified}]
        notes.read(path: string) → {content: string, metadata: object}
        notes.append(path: string, content: string) → {success: bool}  # write scope

        calendar.list_events(from?: date, to?: date, calendar?: string) → [{summary, start, end, location}]

  <security>
    <attack_surface>
      - Strict CSP on login page HTML: default-src 'self'
      - HSTS with preload header on all responses
      - Host header allowlist (reject requests for unknown hosts)
      - Rate limiting on auth endpoints (token bucket per-IP and per-account)
      - cargo audit in CI pipeline
      - CAA DNS records recommended in docs
      - Certificate expiry monitoring as first-class health check
      - No web UI = no XSS/CSRF/session-management attack surface (except Login Flow v2 page)
      - CSRF token on Login Flow v2 form
    </attack_surface>

    <exif_privacy>
      Store EXIF internally for organization and search.
      Strip EXIF by default on any hypothetical public URL (single-user, but defense in depth).
      No full file hashes in URLs — use opaque UUIDs only.
    </exif_privacy>

    <process_isolation>
      ExifTool, libheif, ffmpeg subprocesses:
      - Configurable timeout per tool
      - Resource limits via cgroup where available
      - Crashed decoder cannot crash server
      - Input validation: file type checked by magic bytes before passing to decoder
    </process_isolation>

    <mcp_security>
      - Narrow tools, no URL-fetching capability
      - Scope-restricted tokens + tool allowlist in config (defense in depth)
      - Audit log every call with 90-day retention
      - Constant-time token comparison (subtle crate)
    </mcp_security>

    <backup_security>
      - Server holds only public encryption key; private key stays offline
      - Append-only destination enforcement (B2 Object Lock, S3 Object Lock)
      - Printed-paper recovery code during init
    </backup_security>
  </security>

  <timezone_discipline>
    Every timestamp has a zone. jiff crate throughout.
    Three views available: local-to-event, local-to-viewer, UTC.
    API responses always include UTC offset.
    CLI respects $TZ environment variable.
    CI tests run with multiple TZ values including DST edge cases.
    SQLite stores ISO 8601 with offset (e.g., "2025-06-15T14:30:00-03:00").
  </timezone_discipline>

  <crash_safety>
    - SQLite WAL mode ensures database consistency
    - File writes: atomic tmp → fsync → rename → fsync parent directory
    - Startup scan for orphaned _tmp files in staging directories (clean up)
    - Durable SQLite-backed jobs queue (jobs table) resumes after crashes
    - Chunked upload sessions have expiry; orphaned staging dirs cleaned on startup
  </crash_safety>
